Architecture Notes — How Each Component Maps to Production

▼

SDK / EventPublisher

In production this is a client library (e.g. Java SDK) embedded in services. It generates a UUID per event, checks Redis for idempotency, uses a Circuit Breaker to avoid cascading calls to PubSub, and retries with exponential backoff. The real GCP Pub/Sub client does similar wrapping.

Redis Dedup Cache

A Redis SETNX with TTL is the canonical at-source dedup pattern: write the event UUID on first publish, check on subsequent attempts. TTL eviction prevents unbounded memory growth. In production you'd use Redis Cluster with a 5-minute TTL matching your expected retry window.

Google Cloud Pub/Sub

Pub/Sub guarantees at-least-once delivery — messages can be redelivered after an ack deadline. The simulated failure rate mirrors real outages (quota exhaustion, gRPC errors). In production the SDK calls projects.topics.publish and handles gRPC status codes to decide retry vs. DLQ.

Circuit Breaker (Resilience4j)

Wraps every outbound call to Pub/Sub. A count-based sliding window tracks recent outcomes; when the failure rate crosses 50% the breaker trips OPEN and rejects calls immediately (CallNotPermittedException), preventing thread exhaustion. HALF-OPEN probing validates recovery before reopening traffic.

Dead Letter Queue

Captures events that either exhausted retries or were rejected by an OPEN circuit breaker. In GCP this is a separate Pub/Sub topic configured as the dead-letter-topic on the subscription. A background Cloud Function or worker periodically replays DLQ messages once the circuit closes.

DataFlow Dedup (Layer 2)

A windowed stateful DoFn in Cloud Dataflow maintains a UUID keyed state store with a 2-minute window. Any redeliveries that leaked past Redis (possible during failover or cache TTL edge cases) are caught here before reaching downstream consumers. This is the "defense in depth" dedup layer.

Event Aggregator / Fan-out

Clean events are fanned out to domain sinks — Billing, Audit, Analytics. In production this is a Pub/Sub subscription per sink or a Dataflow pipeline with multiple outputs. Each sink has independent consumer lag tracking via Cloud Monitoring to detect backpressure independently.
